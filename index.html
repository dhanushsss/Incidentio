System health score

Average of per-repo scores (0–100) with a label: HEALTHY, FAIR, or POOR. Roll-up counts include total / utilized / dead endpoints, unresolved outbound calls, stale scans (>7 days), and repos never scanned.

Per-repo cards

Repos sort worst-first. Each card shows health score, primary diagnosis (e.g. orphan ratio, stale scan, unresolved calls), inbound/outbound dependency counts, orphan endpoints, and top sample routes with no callers.

Dead endpoints catalog

Every API route with zero incoming api_calls, across connected repos — method, path, file, line, framework, and how stale the scan is. Sorted so the shakiest data surfaces first.

Authenticated API: GET /api/v1/atlas/dashboard · GET /api/v1/atlas/dead-endpoints — JWT required; computed from existing graph tables (no extra migrations).

Technical reference

Stack, setup, API — expand when you need it

End-to-end in production
  1. Ingest — OAuth, connect repos, deep/incremental scans build endpoint / call / import graph data.
  2. Keep graph fresh — work lives in scan_tasks; workers claim jobs with DB locking (FOR UPDATE SKIP LOCKED) and enforce concurrency limits.
  3. Analyze PR — pull_request webhook triggers async analysis; changed files at PR head join the stored graph.
  4. Compute risk — deterministic verdict and score; comments include confidence breakdown (direct / inferred / unresolved / stale).
  5. Publish — Markdown comment, optional commit status and Slack.
  6. Log prediction — row in pr_predictions for calibration (stored risk normalized to LOW, REVIEW_REQUIRED, BLOCKED).
Languages & stack

Languages (typical): Java (Spring), JavaScript/TypeScript (Express-style), Python (Flask/FastAPI), Ruby, Go — plus fetch/axios-style frontend calls where detected.

Discovery: language-specific extractors (regex-based today; structured so parsers can be swapped) over repo files — quick vs deep scan modes.

Runtime: Java 21, Spring Boot 3.2, PostgreSQL, Flyway, React (Vite). Graph edges include CALLS, IMPORTS, and config links; cross-repo URL matching ties outbound calls to endpoints. Scans use a DB-backed queue; PR-typed jobs are prioritized over manual scans. Multi-instance coordination without Redis/Kafka.

Setup & environment

GitHub OAuth app: Homepage http://localhost:3000 (prod: your UI origin) · Callback must match GITHUB_REDIRECT_URI — default http://localhost:8080/api/auth/callback

Required: GITHUB_CLIENT_ID, GITHUB_CLIENT_SECRET, JWT_SECRET, DB_URL (e.g. jdbc:postgresql://localhost:5432/zerqis), DB_USERNAME, DB_PASSWORD

URLs & CORS: FRONTEND_URL, APP_PUBLIC_URL (public API base; drives webhook URL in GET /api/v1/dashboard/product-config), CORS_ALLOWED_ORIGINS, optional CORS_ALLOWED_ORIGIN_PATTERNS (e.g. Vercel previews).

Frontend (Vite): VITE_API_BASE_URL for a remote API; local dev often uses proxied /api per vite.config.ts. Copy frontend/.env.example → .env when needed.

Other: GITHUB_WEBHOOK_SECRET, SLACK_WEBHOOK_URL (fires when verdict is not SAFE TO MERGE), JWT_EXPIRATION_MS, ZERQIS_VERSION (or ARCHITECT_VERSION fallback → GET /api/public/version), UI_CORE_ONLY, PR flags below. Put Docker secrets in .env beside docker-compose.yml.

Browser token: zerqis_token (legacy architect_token migrates on next sign-in).

API overview

Base URL example http://localhost:8080. Versioned app API under /api/v1. OAuth and webhooks stay unversioned: /api/auth/**, /api/webhooks/**. JWT: Authorization: Bearer <jwt>.

Public: GET /api/auth/github, GET /api/auth/callback, POST /api/webhooks/github, GET /api/public/version, GET /api/public/v1/health, GET /api/public/v1/graph/{owner} with X-API-Key.

Unauthenticated (reference): /actuator/health, /actuator/info — other /api/** requires JWT unless listed in security config.

JWT areas: /api/auth (e.g. GET /me), /api/v1/repos, /api/v1/scan (trigger, status, SSE), /api/v1/graph, /api/v1/insights, /api/v1/impact, GET /api/v1/atlas/dashboard, GET /api/v1/atlas/dead-endpoints, /api/v1/ai, /api/v1/enterprise, /api/v1/keys, /api/v1/dashboard — e.g. GET /api/v1/dashboard/product-config (webhook URL = APP_PUBLIC_URL + /api/webhooks/github), GET /api/v1/dashboard/risky-prs-week.

Scan and rescan triggers return QUEUED while scan_tasks is scheduled.

PR webhook & configuration

POST /api/webhooks/github for pull_request (opened, synchronize). Validate X-Hub-Signature-256 when GITHUB_WEBHOOK_SECRET is set (mismatch → 401). Analysis runs async after 200. Repo must be connected in Zerqis or the webhook is ignored.

Flow: changed files → capped fetch at PR head → join api_calls to endpoints → impact → verdict strings SAFE TO MERGE, REVIEW REQUIRED, BLOCKED + score + confidence breakdown → comment; optional AI Insight block is explanation-only; persist pr_predictions; optional status and Slack; may enqueue full-repo scan as PR-priority.

Testing webhooks: GitHub needs a public URL — use ngrok, Cloudflare Tunnel, or similar; local localhost won’t receive deliveries.

  • PR_MAX_FILES_SCAN (default 50)
  • PR_POST_COMMIT_STATUS — BLOCKED fails status; REVIEW REQUIRED fails only if PR_FAIL_ON_REVIEW=true
  • FRONTEND_URL — links in comments

Repo docs also cover docs/DEPLOYMENT.md (e.g. Vercel + AWS/DO).
