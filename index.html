The Problem

Secrets leak more often than teams realize.

Developers accidentally expose credentials in:

  • application logs
  • CI/CD pipelines
  • Git commits
  • configuration files

Once leaked, attackers can immediately access cloud resources.

Teams struggle to:

  • See all exposures in one place — Logs, Git, and CI are scattered; manual grep and spreadsheets don’t scale.
  • Prioritize by impact — You need to know what an exposed credential can access and what breaks if you rotate it.
  • Remediate quickly — Vault, Kubernetes, and CI need the new value without manual updates.
  • Prove compliance — Auditors and SOC 2 / ISO require an audit trail, evidence export, and clear ownership.

Zerqis gives you one platform to detect, analyze impact, fix, and block future leaks—with audit and compliance built in.

🧠

AI Security Explanation

When a secret is detected, Zerqis automatically generates a clear security analysis.

Example:

Secret Detected: AWS_ACCESS_KEY
Service: payment-service
Environment: production

AI Analysis:
This key belongs to IAM user "payments-prod".

Permissions include:

  • S3 access to payment-logs bucket
  • DynamoDB table transactions
  • Lambda function invoice-generator

Potential Impact:
An attacker could access transaction logs and invoke internal Lambda functions.

Recommended Action:
Rotate the key immediately and update the payment-service environment variables.

Features

Detect → understand impact → fix automatically (and block future leaks). One platform, one API, one dashboard. Not just detection.

🔍

Detection

  • Logs (POST /api/v1/logs), Git push, and pull/merge requests (GitHub + GitLab)
  • CI: GitHub Actions, GitLab CI, CircleCI
  • Regex-based rules: AWS keys, GCP service account keys, Stripe, JWT, passwords, DB URLs, API keys
  • Single inventory: who, where, when, severity
💥

Impact Analysis

  • Blast radius — AWS IAM user and GCP service account roles; permissions and affected resources (S3, DynamoDB, Lambda, GCP) in plain English
  • Usage mapping — Where is this secret used? (services, CI, Terraform, K8s). “What breaks if we rotate?”
  • Breach simulator — “If an attacker uses this key…” can/cannot + data risk; incident severity for SOC
  • Secret lifecycle — Created → Stored → Used → Exposed → Rotated
🔄

Auto-Remediation

  • One-click rotation (AWS and GCP); post-rotation webhook to update Vault, Kubernetes, GitHub
  • Contextual remediation plan by secret type (AWS, GCP, Stripe, DB, JWT); optional AI-generated steps
  • Secret dependency graph — service-to-secret edges; plan updates before rotating
🚫

Block PRs

  • When a secret is found in a PR/MR, set commit status to failure so it can’t be merged
  • GitHub Status API; GitLab supported. Message: “Secret detected. Fix before merging.”
📊

Lifecycle & Migration

  • Secret lifecycle — Full timeline per credential: where it originated, where it’s used, when it leaked, when rotated
  • Migration suggestions — Replace long-lived keys with IRSA (AWS) and Workload Identity (GCP); step-by-step guide with estimated time
📋

Compliance & Audit

  • Audit log (login, API keys, exposures, rotations); evidence export (CSV/JSON)
  • Security Command Center — risk score, critical exposures, trends; Security SLA (MTTR, team risk, executive view); scheduled compliance reports
  • Ownership & accountability — Owner, team, risk owner per secret; assign incidents to owning team; top leaking teams, team risk score. “Who owns this credential? Who rotated it? How fast was it fixed?”
  • SOC 2 / ISO mapping; SIEM webhook
